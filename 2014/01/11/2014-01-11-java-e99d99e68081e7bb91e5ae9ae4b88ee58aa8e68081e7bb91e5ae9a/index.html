Java 静态绑定 与 动态绑定

将一个方法调用和方法主体关联起来被称作绑定。在java中，绑定可以分为静态绑定(前期绑定)和动态绑定(后期绑定)。

1.静态绑定

在程序执行前进行绑定(由编译器或其它连接程序实现)称作静态绑定。

针对java简单的可以理解为程序编译期的绑定。java当中的方法只有final，static，private和构造方法是前期绑定。final方法无法被子类重写Overriding,provate方法对子类不可见，因此他们的静态绑定很容易理解。对于static方法的静态绑定可以看下面的示例程序

public class StaticBindingTest {
    public static void main(String[] args) {
        Base base = new Base();
        Base baseSub = new Sub();
        Sub sub = new Sub();
        Base nullBase = null;
        base.show();//Base
        baseSub.show();//Base
        nullBase.show();//Base
        sub.show();//Sub
    }
}
class Base {
    static void show() {
        System.out.println("Base");
    }
}

class Sub extends Base {
    static void show() {
        System.out.println("Sub");
    }
}

首先需要澄清的是，static方法是否能被重写Overriding。从上面的例子看来好像是static方法能被重写。但是由于static方法是静态绑定的，在类加载的时候其调用已经确定了。所以即使变量baseSub是Sub的实例，但是由于其编译类型是Base，所以通过静态绑定，baseSub调用的是Base的方法。进一步可以看到，即使实例是null，依然可以调用静态方法。更说明了静态方法与实例无关，只和类型相关。如果将子类show方法前的static去掉，则编译器会提示：

This instance method cannot override the static method from Base

2.动态绑定

在运行时根据具体对象的类型进行绑定称作动态绑定。
若一种语言实现了后期绑定，同时必须提供一些机制，可在运行期间判断对象的类型，并分别调用适当的方法。也就是说，编译器此时依然不知道对象的类型，但方法调用机制能自己去调查，找到正确的方法主体。不同的语言对后期绑定的实现方法是有所区别的。但我们至少可以这样认为：它们都要在对象中安插某些特殊类型的信息。动态绑定的过程：

  • 虚拟机提取对象的实际类型的方法表；
  • 虚拟机搜索方法签名；
  • 调用方法。

这里还有一条很重要：对象在调用方法时，方法的签名是在编译期间确定的(重载)，而方法的实现则是在运行时确定的(重写)。看下面的例子：

public class A {
    void show(D o) {
        System.out.println("A D");
    }
    void show(A o) {
        System.out.println("A A");
    }
    public static void main(String[] args) {
        A a1 = new A();
        A a2 = new B();
        B b = new B();
        C c = new C();
        D d = new D();

        a1.show(b);//A A
        a1.show(c);//A A
        a1.show(d);//A D

        a2.show(b);//B A
        a2.show(c);//B A
        a2.show(d);//A D
        a2.show(a2);//B A

        b.show(b);//B B
        b.show(c);//B B
        b.show(d);//A D
                b.show(a2);//B A
    }
}
class B extends A {
    void show(B o) {
        System.out.println("B B");
    }
    void show(A o) {
        System.out.println("B A");
    }
}
class C extends B{}
class D extends B{}

a1的输出很容易理解。a2的输出怎么解释呢？首先a2的编译类型是A，而实际类型是B，当运行

a2.show(b);

时，b的类型是B，所以编译器将在编译期间确定调用方法的签名。A中没有show(B o)，所以编译器确定的方法签名是show(A o)。然后在运行期间，a2的实际类型是B，而B重写了show(A o)，因此运行时调用的将是B的show(A o)。如果B中没有重写show(A o)，那么也不会调用B中的show(B o)，因为编译期间无法搜索到A中的show(B o)，因此最终也只会调用A的show(A o)。当运行

a2.show(d);

时，在A中找到了show(D o)，而且B中没有重写此方法，因此调用的是A中的show(D d)。

b的类型是B，B继承自A，因此B中的方法有继承自A中的A.show(D o)，B中定义的B.show(B o)已经重写A中的B.show(A o)。剩下的分析就简单了。

3.构造器内部的多态方法行为

如果调用构造器内部一个动态绑定的方法，则会用到那个方法被覆盖后的定义。由于这个方法在对象被完全构造前就被调用了，因此会产生一些难以发现的错误。构造器的工作实际上就是创建对象。在任何构造器的内部，整个对象可能只是部分形成—–我们只知道基类的对象已经进行初始化，但却不知道哪些类是从我们这里继承而来的。然而一个动态邦定的方法调用却会向外深入到继承层次结构内部，它可以调用导出类的方法。如果在构造器内部这样做，那么就可能会调用某个方法，而这个方法所操作的成员可能还没有被初始化—这样肯定会导致错误。通过下面的例子我们已可看到错误所在：

class Glyph{
   void draw(){System.out.println("Glyph.draw()");}
   Glyph(){
    System.out.println("Glyph() before draw()");
    draw();
    System.out.println("Glyph() after draw()");
   }
}
class RoundGlyph extends Glyph {
  private int radius = 1;
  RoundGlyph(int r){
    radius = r;
    System.out.println("RoundGlyph.RoundGLyph(), radius = " + radius);
  } 
   void draw(){
       System.out.println("RoundGlyph.draw(), radius = " + radius);
   }
}

public class RolyConstructors {
   public static void main(String[] args){
   new RoundGlyph(5);
   }
}

Glyph.draw()方法设计为将要被覆盖。在RoundGlyph类中我们覆盖了这个方法，但是Glyph构造器会调用这个方法，结构导致了对RoundGlyph.draw()的调用。但是输出的结构显示当Glyph的构造器调用draw()方法时候，radius不是默认初始值1而是0。这是因为：

  • 在其他任何事情发生之前，将分配给对象的储存空间将初始化成二进制的零。
  • 如前面所示那样调用基类的构造器，此时调用被覆盖的draw()方法，由于第一步骤地执行，我们回发现radius的初始值变成了0。
  • 按照申明的顺序调用成员的初始化。
  • 调用导出类的构造器主体。
